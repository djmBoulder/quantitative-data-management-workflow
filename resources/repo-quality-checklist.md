# Repository Quality Checklist

This checklist is for instructors reviewing the IBS CU Boulder **Quantitative Data Management and Workflow for Social Scientists** repository before launch, before internal review, or before a new offering.

Author: Donald Mather, Statistics Consultant, Computing & Research Services, Institute of Behavioral Science, University of Colorado Boulder, Boulder, Colorado

Some checks can be supported by scripts in `code/r/`, but many require human review.

## Automated Checks

- [ ] Run `Rscript code/r/check-course-files.R`.
- [ ] Confirm every module from `00-orientation` through `10-capstone` has the standard files.
- [ ] Run `Rscript code/r/run-r-lab-smoke-tests.R` as a dry run.
- [ ] If ready, run `Rscript code/r/run-r-lab-smoke-tests.R --run`.
- [ ] Review generated smoke-test results in `data/output/`.
- [ ] Confirm generated QA outputs are not committed to GitHub unless intentionally approved.

## Student-Facing Versus Instructor-Facing Separation

- [ ] Student-facing files do not contain answer keys or grading notes.
- [ ] Module `solution_notes.md` files are clearly instructor-facing.
- [ ] Files in `instructor/` are not referenced as student assignment instructions.
- [ ] Student-facing setup guides are in `resources/`.
- [ ] Rubrics are appropriate for students or reviewers to see.

## Stata/R Parallelism

- [ ] Every module includes both `stata_lab.do` and `r_lab.R`.
- [ ] Stata and R labs teach the same core data management task.
- [ ] Stata and R labs use the same synthetic source data where appropriate.
- [ ] Stata and R labs create comparable outputs or diagnostics.
- [ ] Dual-pathway expectations are described as substantively parallel, not line-by-line identical.

## Synthetic Data Only

- [ ] Examples use `synthetic_gss_like` or other approved synthetic/publicly distributable data.
- [ ] No private, restricted, licensed, confidential, or student data are committed.
- [ ] `data/synthetic/` documentation describes the teaching data.
- [ ] Synthetic files can be regenerated from code where applicable.
- [ ] Teaching problems are clearly described as intentional synthetic issues.

## Relative Paths

- [ ] Labs use paths relative to the repository root.
- [ ] Student guides explain the repository root.
- [ ] Assignments discourage absolute paths.
- [ ] Stata and R examples do not rely on one instructor's computer path.
- [ ] Campus lab or remote-system setup notes explain how students should locate the repository.

## Missing File References

- [ ] Referenced data files exist in `data/synthetic/` or are generated by scripts.
- [ ] Referenced output folders are created by scripts or already exist as tracked placeholders.
- [ ] Assignment file names match lab output file names where exact names are required.
- [ ] Syllabus, rubric, and capstone references point to existing files.
- [ ] Links to resources and rubrics use current file names.

## Generated Output Files

- [ ] `.gitignore` excludes `data/working/`, `data/output/`, `logs/`, temporary files, Stata backups, R history files, and system files.
- [ ] Generated files from smoke tests are not staged for commit.
- [ ] Students are told how to submit generated outputs outside GitHub when needed.
- [ ] Scripts do not write generated files into `data/synthetic/`.
- [ ] Capstone instructions distinguish submitted evidence from committed repository files.

## Assessment Alignment

- [ ] Module assignments match module learning objectives.
- [ ] Module checklists prepare students for the capstone.
- [ ] `assignments/capstone-data-management-portfolio.md` matches Module 10 expectations.
- [ ] `rubrics/capstone-rubric.md` evaluates the required capstone evidence.
- [ ] `rubrics/module-checklist-rubric.md` supports formative module review.
- [ ] `syllabus/assessment-plan.md` references current rubric files.

## Badge Competency Alignment

- [ ] `syllabus/badge-competencies.md` lists all core competency areas.
- [ ] Capstone evidence maps to badge competencies.
- [ ] Pathway policy is consistent: Stata, R, or dual Stata/R.
- [ ] One primary pathway is sufficient unless the instructor requires both.
- [ ] Revision triggers match rubric expectations.

## README Clarity

- [ ] Root `README.md` explains the course purpose, audience, Stata/R structure, included materials, and data warning.
- [ ] Each module `README.md` explains purpose, objectives, outputs, skills, pitfalls, time, and capstone relationship.
- [ ] `resources/README.md` indexes student setup and troubleshooting guides.
- [ ] `instructor/README.md` indexes instructor-only implementation materials.
- [ ] Data folder READMEs explain what belongs in each data folder.

## Final Human Review Questions

- [ ] Would a beginner know where to start?
- [ ] Would an instructor know how to prepare a module?
- [ ] Would an internal reviewer understand what the badge certifies?
- [ ] Would a student know whether to submit Stata, R, or dual-pathway evidence?
- [ ] Would a capstone reviewer know what counts as badge-ready?
- [ ] Are there any terms, file names, or policies that conflict across documents?
