# Stata/R Data Wrangling Crosswalk

This crosswalk is a student reference for the IBS CU Boulder microcredential **Quantitative Data Management and Workflow for Social Scientists**. It shows how common data management tasks are handled in Stata and in R/tidyverse workflows using the course's synthetic `synthetic_gss_like` teaching data.

The course supports three pathways:

- Stata pathway: complete graded work in Stata.
- R pathway: complete graded work in R.
- Dual Stata/R pathway: complete both for extra practice or when an instructor requires both.

All students should understand both pathways conceptually. Unless your instructor says otherwise, graded work may be submitted in one primary pathway.

This document is not a full lesson. It is a translation guide. Use it when you recognize the task you need to do but are still learning the syntax in your software pathway.

## Path Note

Paths that begin with `data/synthetic/` refer to tracked synthetic teaching inputs in this repository unless the text says otherwise. Paths that begin with `data/working/`, `data/output/`, or `logs/` are generated by scripts or shown as examples of where outputs should go. If an example path uses a generic name such as `summary.xlsx` or `module-log.txt`, replace it with the filename required by your module assignment.

## 3. Inspecting Data

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| View variable names and types | `describe` | `dplyr::glimpse(person)` or `str(person)` | Shows each variable's name and storage type. Use this immediately after importing. | Do not start recoding until you know whether a variable is numeric, string, labelled, or factor-like. |
| Summarize numeric variables | `summarize age hours_worked household_income` | `summary(person[c("age", "hours_worked", "household_income")])` | Gives a first look at ranges and unusual values. | Special missing codes such as `-8`, `-9`, or `999` can distort summaries if not handled. |
| Tabulate categories | `tabulate gender, missing` | `janitor::tabyl(person$gender)` or `table(person$gender, useNA = "ifany")` | Counts categories and missing values. Useful for variables such as `gender`, `education`, and `smoking_status`. | In R, plain `table()` drops `NA` unless you ask to show missing values. |
| View first rows | `list respondent_id year age gender in 1/10` | `dplyr::slice_head(person, n = 10)` | Lets you inspect actual values, not just summaries. | Do not rely only on the first few rows; they may not show all data problems. |

| --- | --- | --- | --- | --- |
| Keep rows meeting a condition | `keep if age >= 18` | `person_adult <- dplyr::filter(person, age >= 18)` | Filters observations based on a rule. | Create a sample flag before permanently filtering rows in a final workflow. |

## 7. Cleaning Strings And Variable Types

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| Trim spaces | `replace gender = trim(gender)` | `person <- dplyr::mutate(person, gender = stringr::str_trim(gender))` | Removes extra spaces before or after text values. | `"Female"` and `" Female "` look similar to people but not to software. |
| Standardize capitalization | `replace region = lower(trim(region))` | `person <- dplyr::mutate(person, region = stringr::str_to_lower(stringr::str_trim(region)))` | Standard capitalization makes categories easier to match and merge. | Clean the key on both sides before joining files. |
| Convert numeric-looking strings | `destring household_income, replace ignore("$,")` | `person <- dplyr::mutate(person, household_income = readr::parse_number(household_income))` | Converts strings such as `"$45,000"` into numeric values. | Inspect values before converting. Text such as `"DK"` or `"Refused"` needs a rule. |
| Convert numeric categories to labels | `encode education, gen(education_cat)` | `person <- dplyr::mutate(person, education_cat = factor(education))` | Creates a labelled or factor-like categorical variable. | Encoding numbers that are stored as strings can create labels in alphabetical order rather than substantive order. |

## 8. Recoding Variables

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| Recode continuous values into categories | `recode age (18/29=1) (30/44=2) (45/64=3) (65/max=4), gen(age_group)` | `person <- dplyr::mutate(person, age_group = dplyr::case_when(age >= 18 & age <= 29 ~ "18-29", age <= 44 ~ "30-44", age <= 64 ~ "45-64", age >= 65 ~ "65+", TRUE ~ NA_character_))` | Age categories are common in descriptive social science tables. | Check boundary values. A person age 29 or 30 should fall into exactly one category. |
| Recode categories | `generate smoker_current = smoking_status == "Current"` | `person <- dplyr::mutate(person, smoker_current = dplyr::if_else(smoking_status == "Current", 1, 0, missing = NA_real_))` | Creates a derived variable from an existing category. | Decide what to do with missing, refused, or inconsistent source values before coding them as 0. |
| Preserve source meaning | `generate education_clean = education`<br>`replace education_clean = "Graduate" if education == "Grad"` | `person <- dplyr::mutate(person, education_clean = dplyr::recode(education, "Grad" = "Graduate"))` | Derived variables should make the decision visible while keeping the source variable available. | Do not erase the raw category before documenting the recode. |

## 10. Missing Data Checks

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| Check missingness by variable | `misstable summarize age household_income self_rated_health` | `dplyr::summarise(person, dplyr::across(c(age, household_income, self_rated_health), ~ sum(is.na(.))))` | Counts missing values in important variables. | Special codes such as `-8`, `-9`, `999`, or `"DK"` are not missing until recoded. |
| Show missing categories | `tabulate smoking_status, missing` | `table(person$smoking_status, useNA = "ifany")` | Shows whether missing values appear in categorical variables. | R's `NA` is not the string `"NA"`. They behave differently. |
| Recode special numeric missing codes | `mvdecode age household_income, mv(-8 -9 999)` | `person <- dplyr::mutate(person, age = dplyr::na_if(age, -8), age = dplyr::na_if(age, -9))` | Converts special codes into system missing values. | Preserve or document the original meaning of special codes such as refused or don't know. |
| Count row-wise missingness | `egen row_missing = rowmiss(age education household_income self_rated_health)` | `person <- dplyr::mutate(person, row_missing = rowSums(is.na(dplyr::pick(age, education, household_income, self_rated_health))))` | Counts how many core variables are missing for each respondent. | Define which variables count as "core" before using row-wise missingness for sample restrictions. |

## 11. Grouping And Summarizing

## 12. Reshaping Data

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| Check wide-file IDs | `isid respondent_id` | `stopifnot(!anyDuplicated(wide$respondent_id))` | Wide repeated-measures files should have one row per respondent before reshaping long. | Reshape errors often come from duplicate IDs. |
| Reshape wide to long | `reshape long health_ activity_, i(respondent_id) j(year)` | `long <- tidyr::pivot_longer(wide, cols = c(health_2024, health_2025, activity_2024, activity_2025), names_to = c(".value", "year"), names_sep = "_")` | Converts repeated measures stored in columns into person-year rows. | Stubs and suffixes must be named consistently. |
| Reshape long to wide | `reshape wide health_ activity_, i(respondent_id) j(year)` | `wide_again <- tidyr::pivot_wider(long, id_cols = respondent_id, names_from = year, values_from = c(health, activity))` | Converts person-year rows back into one row per respondent. | Check row counts and keys before and after reshaping. |

## 15. Checking Duplicates And Keys

| Task | Stata command or workflow | R/tidyverse workflow | Explanation | Common mistake or caution |
| --- | --- | --- | --- | --- |
| Confirm unique key | `isid respondent_id year` | `stopifnot(!anyDuplicated(person[c("respondent_id", "year")]))` | A key uniquely identifies rows. For person-year data, `respondent_id` plus `year` should usually be unique. | `respondent_id` alone may repeat across years. |
| Report duplicates | `duplicates report respondent_id year` | `dplyr::filter(dplyr::count(person, respondent_id, year), n > 1)` | Shows whether duplicate key values exist. | Duplicate records may be real data issues, intentional repeated measures, or import artifacts. Diagnose before dropping. |

## How To Use This Crosswalk

Use this document when you are translating a workflow idea across software. For example, if you know you need to check duplicate person-year keys, look at the duplicate/key section and compare `isid respondent_id year` with the R `anyDuplicated()` workflow.

For graded assignments, follow your primary pathway unless your instructor requires both. For learning, try reading the other pathway's code and asking: what is the same task? what is different syntax? what data decision is being documented?
