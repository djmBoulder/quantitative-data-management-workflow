# 02 Importing and Inspecting Data Lesson

## Why This Skill Matters

Importing data is the first moment when software makes decisions about your file. It may guess which row contains variable names, whether a column is numeric or text, how missing values are represented, and what to do with spaces or punctuation in names.

If you do not inspect the import carefully, small problems can become large analysis errors. A variable that looks numeric might import as text. A missing-value code such as `-9` might be treated as a real value. A header row might become data. A label in an Excel sheet might become the first observation.

Good data wrangling begins by asking: What kind of file is this, how did it import, and what needs attention before analysis?

## Conceptual Overview

### CSV Files

A CSV file is a plain-text file where values are separated by commas. CSV stands for comma-separated values.

CSV files are common because they are simple and portable. They can be opened by many programs, including Stata, R, Excel, and text editors.

CSV files usually do not store:

- variable labels,
- value labels,
- formatting,
- multiple sheets,
- formulas,
- metadata.

That simplicity is useful, but it means your script and data dictionary need to document what variables mean.

### Excel Files

Excel files, often ending in `.xlsx`, can contain multiple sheets, formatting, formulas, merged cells, and notes outside the data rectangle.

Excel is common in applied research settings, but it requires careful import choices:

- Which sheet should be imported?
- Which row contains variable names?
- Are there notes above or below the data?
- Are numbers stored as numbers or text?
- Did formatting hide values?

For this module, the Excel teaching file contains the same synthetic data as the CSV file. It is generated by `code/r/create-synthetic-data.R` with the `openxlsx` package so instructors can recreate it from code. Later projects may involve messier Excel workbooks.

### Stata `.dta` Files

A Stata `.dta` file is Stata's native data format. It can preserve information that CSV files cannot, including:

- variable labels,
- value labels,
- storage types,
- notes,
- Stata-specific missing values.

`.dta` files are useful for Stata workflows, but they are not a replacement for raw source files or documentation.

### Fixed-Width and Plain-Text Files

Some data arrive as plain-text files that are not comma-separated. A fixed-width file stores values in specific character positions. For example, characters 1 through 5 might be an ID, characters 6 through 7 might be age, and characters 8 through 9 might be state.

Fixed-width files usually require a layout file or codebook. Without that documentation, it is hard to know where one variable ends and the next begins.

This module does not ask you to import a fixed-width file. You only need to recognize why fixed-width data require different instructions from CSV or Excel files.

## Importing Without Editing Raw Data

The raw files for this module are:

- `data/synthetic/synthetic_gss_like_person_2024.csv`
- `data/synthetic/synthetic_gss_like_person_2024.xlsx`

Do not open these files and manually fix them. If a name is messy or a value looks strange, document it and use code to make a working copy.

The safe workflow is:

1. Import the raw file.
2. Inspect the imported data.
3. Standardize names in code.
4. Save a working copy.
5. Leave the raw file unchanged.

## Inspecting After Import

After importing data, always inspect:

- number of rows,
- number of columns,
- variable names,
- variable storage types,
- labels or absence of labels,
- missing values,
- unusual category spelling or capitalization,
- values that do not match the data dictionary.

The `synthetic_gss_like` data intentionally include teaching problems such as:

- headers with spaces and punctuation,
- inconsistent capitalization in categories,
- special missing codes like `-8`, `-9`, `99`, and `999`,
- numeric-looking variables stored as strings,
- income values with dollar signs and commas,
- survey weights with comma decimal strings.

At this stage, your goal is to identify these issues, not fully clean them.

## Standardizing Variable Names

Raw file names often arrive with spaces, punctuation, and inconsistent capitalization:

```text
Respondent ID
Survey Year
Race / ethnicity
Household Income
survey weight
```

For coding, it is easier to use a consistent style:

```text
respondent_id
year
race_ethnicity
household_income
survey_weight
```

In this course, use lowercase snake case:

- all lowercase letters,
- underscores between words,
- no spaces,
- no punctuation,
- names that describe the variable clearly.

The data dictionary in `data/synthetic/data-dictionary.md` lists the recommended clean names.

## Stata Workflow

In Stata, use `import delimited` for CSV files:

```stata
* Import the CSV file from a relative path.
import delimited using "data/synthetic/synthetic_gss_like_person_2024.csv", clear varnames(1) case(lower)
```

Use `import excel` for Excel files:

```stata
* Import the first row as variable names from the Excel file.
import excel using "data/synthetic/synthetic_gss_like_person_2024.xlsx", clear firstrow case(lower)
```

After import, inspect the data:

```stata
describe
codebook age
misstable summarize
tabulate gender, missing
```

Then standardize names and save a working file:

```stata
rename survey_year year
save "data/working/synthetic_gss_like_person_2024_imported.dta", replace
```

## R Workflow

In R, use `read.csv()` for CSV files:

```r
# Read the CSV file from a relative path.
survey_csv <- read.csv("data/synthetic/synthetic_gss_like_person_2024.csv",
                       check.names = FALSE,
                       stringsAsFactors = FALSE)
```

Use `readxl::read_excel()` for Excel files:

```r
# Read the first sheet of the Excel file.
survey_excel <- readxl::read_excel("data/synthetic/synthetic_gss_like_person_2024.xlsx")
```

After import, inspect the data:

```r
names(survey_csv)
str(survey_csv)
summary(survey_csv)
colSums(is.na(survey_csv))
```

Then standardize names and save working files:

```r
names(survey_csv) <- clean_names(names(survey_csv))
saveRDS(survey_csv, "data/working/synthetic_gss_like_person_2024_imported.rds")
```

## Common Import Problems

Watch for:

- wrong working directory,
- missing source file,
- headers imported as data,
- blank columns,
- unexpected variable names,
- numeric-looking values imported as strings,
- special missing codes treated as real values,
- Excel sheets with notes or formatting,
- category values that differ only by capitalization or spaces.

When you find a problem, write it down. Later modules will show how to clean and validate those issues.
